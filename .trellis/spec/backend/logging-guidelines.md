# Logging Guidelines

> Logging and runtime diagnostics conventions.

---

## Overview

Backend modules use the standard `logging` library. Importable modules define a module logger and leave handler configuration to scripts. CLI entry points configure a simple INFO console format, while selected long-running workflows also write files under `outputs/logs/`.

```python
logger = logging.getLogger(__name__)
```

Script entry points commonly use:

```python
logging.basicConfig(level=logging.INFO, format="%(levelname)s:%(name)s:%(message)s")
logger = logging.getLogger(__name__)
```

---

## Log Levels

| Level | Use |
| --- | --- |
| `INFO` | Stage starts/completions, input summaries, output paths, optimization progress, cache reuse |
| `WARNING` | Non-fatal issues that need attention, skipped symbols, unknown config keys, fallback dates |
| `ERROR` / `logger.exception` | Workflow failure with traceback and prior context |
| `DEBUG` | Rarely used; avoid adding noisy debug logs to committed code |

Examples:

- `src.config_loader._warn_unknown_config_keys` logs unknown config keys as warnings but keeps compatibility.
- `src.signal_generator._effective_signal_date` warns when falling back to an earlier factor date.
- `scripts/run_backtest.py` logs input summaries before work and calls `logger.exception` on failure.

---

## File Logs

`scripts/run_backtest.py` creates a per-run log file:

```text
outputs/logs/backtest_YYYYMMDD_HHMMSS_microseconds.log
```

The helper `_configure_run_logging(out_dir)` removes the previous backtest file handler before adding a new one. Preserve the `_quant_box_backtest_file` marker and the test coverage in `tests/test_run_backtest_logging.py` if this behavior changes.

Background auto-signal jobs write logs under:

```text
outputs/logs/auto_signal_*.log
```

History backfill uses:

```text
outputs/logs/history_data_backfill.log
```

## Status Artifacts Are Part Of Diagnostics

For workflows with multiple stages, JSON status files are as important as logs:

- `outputs/auto_run_status.json` is updated after each `_stage(...)`.
- `outputs/data_update_progress.json` tracks resumable raw-data updates.
- `outputs/auto_signal_job.json` tracks supervised background jobs.
- `outputs/auto_run_metrics.prom` is generated by `scripts/export_auto_status_metrics.py`.

When adding a new long-running workflow, write a status artifact that records stage name, state, timestamp, and a short message.

---

## What To Log

Log compact summaries, not raw data:

- Start/end dates and requested file paths.
- DataFrame shape, date range, instrument count, field list.
- Config snapshots after redaction/safe filtering.
- Quality gates and block reasons.
- Output file paths.
- Retry/skip counts and failed-symbol file path.

Example from `scripts/run_backtest.py`:

```python
logger.info("Price input: %s", json.dumps(price_summary, ensure_ascii=False, default=str))
logger.info("Output files: equity=%s holdings=%s trades=%s metrics=%s", equity_path, holdings_path, trades_path, metrics_path)
```

---

## What Not To Log

Do not log:

- Tushare tokens.
- Full proxy URLs containing credentials.
- `config/settings.local.yaml` contents.
- Account holdings files or broker/account identifiers beyond explicit report outputs.
- Full DataFrames from raw market caches.

Use `TushareHttpClient.redacted_request_preview()` and `describe_endpoint()` for Tushare diagnostics.

---

## Print vs Logging

Use logging in importable modules and operational scripts. `print` is acceptable for small terminal-report tools and one-off research/probe scripts where stdout is the product, such as:

- `scripts/show_update_progress.py`
- `scripts/run_goal_audit.py`
- `scripts/benchmark_scoring.py`
- selected research/probe scripts that print a saved output path.

Do not mix large `print` dumps into long-running automated workflows that already use `logger` and status files.

---

## Testing Logging

Use `self.assertLogs(...)` when warning/error behavior is part of the contract. Existing examples:

- `tests/test_config_loader.py` asserts unknown config keys warn without rejecting the config.
- `tests/test_run_backtest_logging.py` asserts the backtest file handler is replaced instead of duplicated.

If a log message is used by users to diagnose a workflow, test the behavior around it rather than relying only on manual inspection.
